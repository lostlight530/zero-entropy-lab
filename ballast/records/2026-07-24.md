# Ballast 日记录 2026-07-24

## 研究问题

Agent 对结果未知的非幂等操作进行恢复时, 盲重试、仅按请求键去重、以及请求键绑定当前意图并在重试前回读结果, 哪条路径能够阻止重复副作用和错误复用旧结果.

## 来源依据

查询日期为 2026-07-24.

- [RFC 9110 第 9.2.2 节](https://www.rfc-editor.org/rfc/rfc9110.html#section-9.2.2), 发布于 2022 年 6 月, 说明响应读取前通信失败时可以重试已知幂等请求, 非幂等请求只有在已知语义幂等或能确认原操作未应用时才应自动重试.
- [Amazon EC2 API 幂等文档](https://docs.aws.amazon.com/ec2/latest/devguide/ec2-api-idempotency.html), 在线文档未显示稳定更新日期, 说明相同令牌与相同参数的成功重试不执行额外动作, 相同令牌与不同参数应返回参数不匹配.
- [Microsoft Retry pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/retry), 在线文档未显示稳定更新日期, 说明服务可能已经成功处理请求但响应发送失败, 非幂等重试会产生重复副作用.

RFC 约束客户端重试语义, AWS 给出令牌和参数绑定合同, Microsoft 描述确认丢失故障. 三者没有发现实质冲突. 它们不证明本地模拟等同于真实远端服务.

## 可证伪假设

路径 A 在确认丢失后直接重试, 预计产生两次副作用.

路径 B 只按请求键去重, 预计同意图重放只有一次副作用, 但相同键承载不同意图时会错误返回旧结果.

路径 C 在收据中绑定规范化意图摘要, 重试前回读当前结果, 相同意图重放返回旧结果且零写入, 不同意图复用旧键时安全停止.

如果路径 A 没有重复副作用, 路径 B 能正确拒绝不同意图, 或路径 C 产生第二次副作用、接受意图冲突或未通过独立验证, 对应假设被推翻.

## 控制条件

- 三条路径使用相同请求键、原始意图和确认丢失位置.
- 原始意图为对受控资源执行数量 4 的保留操作, 冲突意图只把数量改为 7.
- 故障发生在副作用和收据已经写入之后、客户端收到确认之前.
- 每条路径使用隔离的效果清单和收据文件.
- 验证器在独立 Python 进程中回读文件, 重新规范化当前意图并计算 SHA-256.
- 所有样本使用 ballast 内固定前缀, 开始前不得存在, 结束后逐文件删除.

## 实验设计

1. 路径 A 提交原始操作并丢失确认, 客户端不读取状态而再次提交, 然后验证副作用数量.
2. 路径 B 提交后使用同一键重放原始意图, 再用同一键提交数量 7 的新意图, 分别验证原始与当前意图.
3. 路径 C 提交后先由独立验证器读取当前结果, 已确认原操作生效时不重新提交.
4. 对路径 C 再执行同输入重放和不同意图复用旧键, 检查写入数、拒绝状态和最终原始效果.
5. 对六项预期状态执行断言并清理全部受控文件.

## 原始观测

状态轨迹为 提交副作用 → 确认丢失 → 恢复路径分叉 → 独立回读 → 重放或安全停止.

- 路径 A 调用 2 次, 应用副作用 2 次, 状态文件写入 4 次. 验证器以 effect_count_mismatch 拒绝, 退出状态为 1.
- 路径 B 调用 3 次, 应用副作用 1 次, 状态文件写入 2 次. 相同意图重放返回 duplicate_returned, 原始意图验证通过.
- 路径 B 使用同一键提交数量 7 时仍返回 duplicate_returned. 当前意图验证以 intent_digest_mismatch 和 effect_intent_mismatch 拒绝, 退出状态为 1.
- 路径 C 首次写入后, 重试前回读验证通过, 因而重新提交次数为 0.
- 路径 C 同输入重放返回 duplicate_returned 且写入 0 次, 不同意图返回 intent_mismatch 且写入 0 次.
- 路径 C 最终只有 1 次副作用, 原始意图验证通过, 退出状态为 0.
- 六项机制断言全部为真, 有效完成为真, 受控状态完成清理.
- 第一轮验证后有效耗时为 429.739 毫秒.
- 提交前复验得到相同路径状态、操作数、验证结果与清理结果, 验证后有效耗时为 373.475 毫秒.

实验准备阶段曾发生两项补丁传输问题. 截断的实验文件被编译检查拒绝, 空验证器虽然退出状态为 0 但没有输出, 被实验的输出断言拒绝. 两者都在正式机制实验前修复并完成清理, 不计入路径证据.

## 独立验证

verify_unknown_outcome.py 在独立 Python 进程中回读效果清单和收据, 检查指定请求键的效果数量恰好为 1、当前意图摘要、效果意图和结果标识一致. 验证期望来自调用时的当前意图, 不读取生产函数返回的成功状态.

独立性限制是生产器与验证器仍共享 JSON 合同、SHA-256 和规范化规则. 两者也运行在同一文件系统, 不能排除共同的存储或解析错误.

## 强反例

路径 B 在同一请求键承载数量 7 的新意图时没有产生重复效果, 看似满足去重和低操作数目标, 但它返回了数量 4 的旧结果并报告 duplicate_returned. 只有绑定当前意图的验证才将其拒绝.

这个反例推翻了请求键存在且副作用数量为 1 就足以证明重试正确的解释.

## 路径比较

- 路径 A 操作最直接, 但 2 次调用产生 2 次副作用, 结果质量失败.
- 路径 B 将副作用降为 1 次, 同意图恢复正确, 但不同意图复用旧键时产生假成功.
- 路径 C 首次副作用 1 次, 重试前回读阻止重新提交, 同输入重放零写入, 意图冲突安全停止, 最终验证通过.
- 三条路径没有分别计时, 因此不能断言路径 C 的速度优势. 整体耗时包含独立进程验证与清理.
- 路径 B 与 C 的低写入数相同, 但只有路径 C 保持当前意图质量, 说明操作数相同不代表结果等价.

## 暂时结论

状态为观察.

在本次受控结果未知故障中, 盲重试造成重复副作用, 单独请求键只能处理同意图重放, 不能识别同键不同意图. 请求键绑定意图摘要并在重试前回读已应用结果, 同时阻止了重复副作用和旧结果误用.

该结果尚未升级为候选. 本地双文件模拟不具有真实远端并发、持久化和网络语义.

## 复验条件

在另一时间窗口改变副作用类型或确认丢失位置, 加入两个并发执行者竞争同一请求键. 验证收据写入前后中断、收据与副作用不同步、令牌过期和结果查询暂时不可用时是否仍能安全停止.

使用不同实现或不同规范化规则的验证器攻击共享假设.

## 事实分层

### 已验证事实

盲重试产生两次副作用. 仅按键去重阻止同意图重复, 却接受同键不同意图并返回旧结果. 意图绑定路径在回读后未重新提交, 同输入重放零写入, 意图冲突安全停止, 最终独立验证通过并完成清理.

### 基于证据的推断

结果未知恢复需要同时识别操作身份与当前意图. 在能够可靠查询已应用结果的条件下, 重试前回读比直接重放更能降低重复副作用.

### 未验证事项

真实远端 API、并发竞争、收据与副作用非原子写入、请求键过期、不可查询结果、不可逆操作和不同实现验证器均未覆盖.

## 指标

- 有效完成: 路径 C 为是, 路径 A 与路径 B 冲突意图为否.
- 重放一致与幂等通过: 路径 C 为是.
- 中断恢复: 路径 C 在确认丢失后恢复.
- 假成功: 路径 B 冲突意图 1 次, 独立验证拦截 1 次.
- 重复副作用: 路径 A 1 次额外效果.
- 无效操作: 正式协议内 0.
- 正式重试或重放: 路径 A 1 次, 路径 B 2 次, 路径 C 同输入重放 1 次与冲突探测 1 次.
- 人工介入: 0.
- 状态文件写入: 路径 A 4 次, 路径 B 2 次, 路径 C 2 次.
- 验证后的有效耗时: 第一轮 429.739 毫秒, 提交前复验 373.475 毫秒.
